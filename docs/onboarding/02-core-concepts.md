# 核心概念

理解这些核心概念，将帮助你更好地使用 Facio Superpowers。

## 三大核心概念

```
┌─────────────────────────────────────────────────┐
│                Facio Superpowers                │
├─────────────────┬───────────────┬───────────────┤
│     Skill       │    Context    │     Task      │
│   (工作方法)     │   (讨论上下文) │    (开发任务)  │
├─────────────────┼───────────────┼───────────────┤
│ 告诉 AI 怎么做   │ 记录想法和决策 │ 跟踪实现进度   │
│ 如：TDD、设计评审│ 如：需求讨论   │ 如：开发某功能 │
└─────────────────┴───────────────┴───────────────┘
```

---

## Skill（技能）

### 是什么？

Skill 是 **AI 的工作方法论**。它告诉 AI：
- 做某类任务时应该遵循什么流程
- 什么时候该问什么问题
- 如何确保质量

### 为什么需要 Skill？

没有 Skill 的 AI 像一个"听话但没经验的实习生"——你说什么它做什么，但不知道最佳实践。

有了 Skill 的 AI 像一个"懂规矩的高级工程师"——不仅能做事，还知道：
- 写代码前要先设计
- 改代码前要先理解
- 完成后要验证和文档

### 核心 Skill 一览

| Skill | 作用 | 何时触发 |
|-------|------|---------|
| `brainstorming` | 需求讨论、方案设计 | 说"我想做 xxx"时 |
| `writing-plans` | 把设计拆成可执行任务 | 设计确认后 |
| `test-driven-development` | 先写测试再写代码 | 开始实现时 |
| `systematic-debugging` | 系统化排查问题 | 遇到 bug 时 |
| `verification-before-completion` | 完成前验证质量 | 说"做完了"时 |
| `prepare-context` | 查找相关文档和决策 | 开始开发前 |

### 如何使用？

**方式一：直接调用**
```
/brainstorming
/prepare-context
```

**方式二：自动触发**

AI 会根据你说的话自动判断该用哪个 Skill：
- "我想加个功能" → 触发 `brainstorming`
- "这里有 bug" → 触发 `systematic-debugging`
- "开发完了" → 触发 `verification-before-completion`

## Context（上下文）

### 是什么？

Context 是 **讨论的容器**，用于记录：
- 需求想法的背景
- 讨论过程中的分析
- 最终的决策和原因
- 验收标准

### 为什么需要 Context？

| 没有 Context | 有 Context |
|-------------|-----------|
| 讨论内容散落在各种聊天工具里 | 讨论集中记录，可追溯 |
| 换个人接手不知道之前讨论了什么 | 新人看 Context 就能了解背景 |
| 需求变更不知道为什么当初这么设计 | 决策原因清楚记录 |
| 验收时扯皮"我以为是这样" | 验收标准提前明确 |

### Context 的生命周期

```
open（讨论中）
   ↓
decided（已决策）
   ↓
claimed（已认领）
   ↓
closed（已完成）
```

### 如何使用？

**创建 Context**：当你说"我想做 xxx"、"有个想法"时，AI 会询问是否创建。

**追加讨论**：在讨论过程中，有价值的内容会自动记录。

**记录决策**：当你说"就这样吧"、"确定用方案 A"时，AI 会记录决策。

**查看 Context**：
```
帮我看看收藏夹功能的讨论
```

## Task（任务）

### 是什么？

Task 是 **具体的开发工作项**，从 Context 的决策中产生。

### Context 与 Task 的关系

```
Context: "用户希望有收藏夹功能"
    │
    ├─→ 讨论：需要收藏什么？文章？商品？
    ├─→ 讨论：如何展示收藏列表？
    ├─→ 决策：收藏文章，列表按时间排序
    │
    └─→ 生成 Tasks:
        ├─ Task 1: 设计收藏数据表
        ├─ Task 2: 实现收藏 API
        ├─ Task 3: 实现收藏列表页面
        └─ Task 4: 编写测试用例
```

### Task 的状态

| 状态 | 含义 |
|-----|------|
| 待开发 | 等待开始 |
| 开发中 | 正在进行 |
| 待测试 | 开发完成，等待测试 |
| 待验收 | 测试通过，等待产品验收 |
| 已完成 | 全部完成 |

## 两个工作阶段

Facio Superpowers 将工作分为两个阶段，AI 在不同阶段扮演不同角色。

### Context 阶段（需求讨论）

**AI 角色**：产品经理

**AI 会做**：
- 追问需求细节
- 识别边界条件
- 提出验收标准
- 生成测试用例

**AI 不会做**：
- 讨论技术实现细节
- 建议具体技术方案

### Task 阶段（开发实现）

**AI 角色**：研发专家

**AI 会做**：
- 分析技术方案
- 编写代码
- 考虑性能和安全
- 更新任务进度

**AI 不会做**：
- 修改已确定的需求
- 跳过测试直接完成

### 阶段切换

| 你说... | 触发 |
|--------|------|
| "开始开发" | 从 Context → Task 阶段 |
| "需求要改" | 从 Task → 新 Context（followup） |
| "做完了" | Task 状态更新 |

## 工作流程全景

```
        ┌──────────────────────────────────────────────────────────┐
        │                    完整工作流程                           │
        └──────────────────────────────────────────────────────────┘
                                    │
        ┌───────────────────────────┼───────────────────────────┐
        │      Context 阶段         │       Task 阶段            │
        │      (AI = 产品经理)       │       (AI = 研发专家)       │
        ├───────────────────────────┼───────────────────────────┤
        │                           │                           │
        │  "我想做 xxx"              │  "开始开发"                │
        │       ↓                   │       ↓                   │
        │  创建 Context             │  认领 Context → Task      │
        │       ↓                   │       ↓                   │
        │  讨论需求细节              │  /prepare-context         │
        │       ↓                   │       ↓                   │
        │  确定验收标准              │  brainstorming → 技术方案  │
        │       ↓                   │       ↓                   │
        │  生成测试用例              │  writing-plans → 任务拆分  │
        │       ↓                   │       ↓                   │
        │  记录决策                  │  TDD 实现                 │
        │                           │       ↓                   │
        │                           │  verification → 验证完成   │
        │                           │       ↓                   │
        │                           │  更新 Task 状态            │
        │                           │                           │
        └───────────────────────────┴───────────────────────────┘
```
