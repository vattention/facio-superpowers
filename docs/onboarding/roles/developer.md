# 开发者指南

本指南帮助开发者了解如何使用 Facio Superpowers 进行日常开发工作。

## 开发者的核心工作流

```
┌─────────────────────────────────────────────────────┐
│                   开发工作流                         │
└─────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────┐
│ /prepare-context│ ← 查找相关文档和决策
└────────┬────────┘
         ▼
┌─────────────────┐
│ 认领任务/需求    │ ← 从 Context 或 Task 开始
└────────┬────────┘
         ▼
┌─────────────────┐
│ /brainstorming  │ ← 技术方案讨论
└────────┬────────┘
         ▼
┌─────────────────┐
│ writing-plans   │ ← 任务拆分（自动）
└────────┬────────┘
         ▼
┌─────────────────┐
│ TDD 实现        │ ← 先测试后代码
└────────┬────────┘
         ▼
┌─────────────────┐
│ /verification   │ ← 完成前验证
└─────────────────┘
```

---

## 一、开始开发前

### 1. 准备上下文（必做）

```
/prepare-context
```

AI 会搜索并展示：
- 相关的 ADR（架构决策记录）
- 历史讨论 Context
- 相关设计文档

**为什么重要**：
- 避免违反已有的技术决策
- 了解之前讨论过什么
- 知道有哪些可复用的代码

### 2. 认领任务

**从 Context 认领**（已有需求讨论）：
```
我来做收藏夹功能
```

**从 Task 认领**（已有具体任务）：
```
我认领"实现搜索 API"这个任务
```

**查看可认领的任务**：
```
有哪些待认领的任务？
```

---

## 二、技术方案设计

### 进入技术讨论

认领后，AI 会自动引导你进行技术方案讨论。

AI 会问：
- "打算用什么技术栈实现？"
- "数据结构怎么设计？"
- "API 接口怎么定义？"

### 方案选择

AI 通常会提供 2-3 个方案：

```
方案 A：使用 localStorage 存储收藏
- 优点：简单，无需后端
- 缺点：不能跨设备同步

方案 B：使用后端 API 存储
- 优点：跨设备同步
- 缺点：需要开发 API

方案 C：先 localStorage，后期加同步
- 优点：快速上线，渐进增强
- 缺点：需要处理数据迁移

推荐方案 B，因为...
```

选择方案：
```
用方案 B
```

## 三、任务拆分

### 自动生成任务

确认方案后，AI 会使用 `writing-plans` 自动拆分任务：

```
任务拆分：
1. [5min] 创建收藏表的数据库 migration
2. [10min] 实现收藏 API - POST /api/favorites
3. [10min] 实现取消收藏 API - DELETE /api/favorites/:id
4. [10min] 实现获取收藏列表 API - GET /api/favorites
5. [15min] 实现收藏按钮组件
6. [15min] 实现收藏列表页面
7. [10min] 编写 API 测试
8. [10min] 编写组件测试
```

### 调整任务

如果觉得拆分不合理：
```
把任务 2、3、4 合并成一个"实现收藏 API"
```

## 四、编码实现（TDD）

### TDD 流程

AI 默认采用测试驱动开发（TDD）：

**Red**：先写失败的测试
```typescript
// AI 会先生成测试
describe('Favorites API', () => {
  it('should add item to favorites', async () => {
    const response = await request(app)
      .post('/api/favorites')
      .send({ itemId: '123' });
    expect(response.status).toBe(201);
  });
});
```

**Green**：写最小代码让测试通过
```typescript
// AI 再实现代码
app.post('/api/favorites', async (req, res) => {
  const { itemId } = req.body;
  await db.favorites.create({ itemId, userId: req.user.id });
  res.status(201).json({ success: true });
});
```

**Refactor**：重构优化

### 跳过 TDD（不推荐）

如果确实是简单改动：
```
这个很简单，直接写代码不用写测试
```

但 AI 会提醒你测试的重要性。

## 五、代码评审

### 发起评审

完成开发后：
```
帮我 review 一下这次改动
```

AI 会检查：
- 代码风格
- 潜在 bug
- 性能问题
- 安全问题

### 收到评审意见

```
/receiving-code-review
```

AI 会帮你：
- 理解评审意见
- 判断建议是否合理
- 实施修改

---

## 六、完成验证

### 验证命令

```
/verification-before-completion
```

AI 会：
1. 运行所有测试
2. 检查代码覆盖率
3. 检查是否需要生成 ADR
4. 更新相关文档

### 更新任务状态

```
收藏 API 开发完成了
```

AI 会更新 Task 状态为"待测试"。

---

## 七、调试（修复 Bug）

### 描述问题

```
用户收藏后刷新页面，收藏状态丢失了
```

AI 进入 `systematic-debugging` 模式：

**Phase 1: 收集信息**
- 复现步骤
- 错误日志
- 浏览器控制台输出

**Phase 2: 形成假设**
- 可能是状态没有持久化
- 可能是获取状态的 API 调用失败
- 可能是组件初始化时序问题

**Phase 3: 验证假设**
- 检查 API 调用日志
- 检查数据库记录
- 添加调试日志

**Phase 4: 修复验证**
- 修改代码
- 添加防回归测试
- 确认问题解决

## 九、最佳实践

### 1. 永远先准备上下文

开发前花 1 分钟跑 `/prepare-context`，避免：
- 重复实现已有功能
- 违反已有架构决策
- 错过可复用的代码

### 2. 小步提交

每完成一个小任务就提交：
```
git commit -m "feat: 实现收藏 API"
```

而不是做完所有功能才提交一次大的。

### 3. 测试先行

即使 AI 没有主动写测试，也要求它写：
```
先写测试
```

测试是最好的文档和防护网。

### 4. 有疑问就问

不确定的地方直接问 AI：
```
这里用 useState 还是 useReducer 更好？
```

AI 会给出分析和建议。

### 5. 发现需求问题及时反馈

如果开发过程中发现需求有问题：
```
这个需求有问题，xxx 情况没考虑到
```

AI 会创建 followup Context，让产品知道需要补充讨论。

## 十、常见问题

### Q: AI 生成的代码风格和项目不一致

```
请按照项目的代码风格重写
```

或在项目 CLAUDE.md 中配置代码风格规范。

### Q: AI 拆分的任务太细/太粗

```
任务粒度太细了，合并一下
任务太大了，再拆细一点
```

### Q: 想用自己的实现方式

```
我想用 xxx 方式实现，不用你建议的方案
```

AI 会尊重你的选择，但可能会提醒潜在风险。

### Q: 测试老是过不了

```
帮我分析一下测试为什么失败
```

AI 会分析失败原因并给出修复建议。
