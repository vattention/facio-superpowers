## 场景一：开发新功能

### 完整流程

#### 1. 准备上下文

```
/prepare-context
```

AI 会自动搜索：
- 相关的架构决策记录（ADR）
- 历史讨论的 Context
- 相关设计文档

**为什么重要**：避免重复造轮子，避免违反已有的技术决策。

#### 2. 需求讨论（Context 阶段）

告诉 AI 你想做什么：

```
我想给用户列表页加个搜索功能
```

AI 会：
1. 询问是否创建 Context → 回答"是"
2. 追问需求细节：
   - 搜索哪些字段？
   - 实时搜索还是点击按钮搜索？
   - 搜索结果如何高亮？
3. 确定验收标准
4. 生成测试用例

**关键动作**：
- 认真回答 AI 的问题，越清楚后面越顺利
- 确认验收标准时要仔细，这是最终验收的依据

#### 3. 确认开始开发

```
验收标准确认，开始开发
```

AI 会：
1. 记录决策
2. 进入 Task 阶段
3. 开始技术方案讨论

#### 4. 任务拆分（自动）

AI 会使用 `writing-plans` Skill 把功能拆成小任务：

```
任务拆分：
1. 添加搜索输入框组件
2. 实现搜索 API 接口
3. 添加防抖处理
4. 实现搜索结果高亮
5. 编写单元测试
```

#### 5. 编码实现

AI 会按任务顺序实现，每个任务：
- 先写测试（TDD）
- 再写实现
- 测试通过后继续下一个

**你需要做**：
- 审查 AI 生成的代码
- 有问题及时指出
- 确认每个任务的完成

#### 6. 验证完成

```
/verification-before-completion
```

AI 会：
- 运行所有测试
- 检查代码质量
- 生成/更新文档
- 确认验收标准是否满足

## 场景二：修复 Bug

### 完整流程

#### 1. 描述问题

```
用户登录后跳转到首页，但有时候会显示未登录状态
```

#### 2. AI 进入调试模式

AI 会使用 `systematic-debugging` Skill：

**Phase 1: 信息收集**
- 复现步骤是什么？
- 是否有报错信息？
- 什么情况下会出现？

**Phase 2: 假设形成**
- 可能是 token 存储问题
- 可能是异步时序问题
- 可能是缓存问题

**Phase 3: 验证假设**
- 添加日志确认
- 检查相关代码
- 逐一排除

**Phase 4: 修复验证**
- 修改代码
- 编写测试防止回归
- 确认问题解决

#### 3. 更新任务

如果这个 Bug 关联了 Task：

```
这个 bug 修好了
```

AI 会自动更新 Task 状态。

## 场景三：代码评审

### 作为评审者

```
帮我 review 一下这个 PR：[PR 链接]
```

AI 会检查：
- 代码风格是否符合规范
- 是否有潜在的 bug
- 测试覆盖是否充分
- 是否符合架构决策

### 收到评审意见后

```
/receiving-code-review
```

AI 会帮助你：
- 理解评审意见
- 评估修改建议是否合理
- 实施修改

## 场景四：版本规划（产品经理）

### 创建新版本

```
创建 video-editor 的 v0.5 版本，目标是完成基础剪辑功能
```

AI 会：
1. 创建版本目录
2. 列出可规划的 Context
3. 询问要加入哪些需求

### 查看版本进度

```
video-editor v0.5 的进度怎么样了？
```

AI 会展示：
- 已完成的需求
- 进行中的任务
- 阻塞的问题

## 常见问题

### Q: AI 问太多问题了，我能跳过吗？

可以说"先按默认的来"或"这些细节后面再定"，但建议尽量回答——前期多花 5 分钟讨论，后期能省 5 小时返工。

### Q: 我想直接写代码，不想走流程

可以说"简单改动，直接实现"，AI 会简化流程。但对于复杂功能，流程能帮你避免很多坑。

### Q: Context 和 Task 太多记不住怎么办？

```
我现在有哪些进行中的任务？
最近讨论的 Context 有哪些？
```

AI 会帮你查询。

### Q: 讨论到一半要离开怎么办？

直接离开就好。Context 已经记录了讨论内容，下次回来说：

```
继续讨论收藏夹功能
```

AI 会从上次的地方继续。
